MVP / 着手前 Overview

🥗 MuveEats

「動く」＋「食べる」をもじったヘルスケアアプリ。トレーニング記録と食事管理を 一つのダッシュボードで扱い、食事写真からはカロリー・PFCを自動推定します。

v0.1 draft
2026-05-25

1. プロジェクトのゴール

「今日どれだけ動いて、どれだけ食べたか」を、面倒な入力なしに振り返れる状態を作る。 食事写真をスマホで撮るだけでカロリー/PFCを推定し、手動記録と統合する。

主軸価値
入力の摩擦を最小化

写真1枚 → 推定 → 1タップ保存。

ターゲット
1人ユース（まずは自分）

多人数化は認証導入後。

差別化
運動 × 食事を一体管理

消費 / 摂取の両軸ビューを志向。

2. MVP スコープ

機能 内容 状態
食事写真 → カロリー解析 Gemini に画像を送り、料理名 / kcal / P・F・C / 自信度を JSON で取得 スキャフォルド済
食事の手動入力 料理名・栄養素・日時・メモを記録、履歴を一覧 スキャフォルド済
ダッシュボード / サマリー 今日・直近7日のカロリーと PFC、記録件数を集計 スキャフォルド済
認証 (Supabase Auth) メール or Google OAuth。RLS は導入済みのため繋ぎ込みのみ 次着手
トレーニング記録 種目 / セット / 重量 / rep の記録と履歴 フェーズ2
写真のストレージ保存 Supabase Storage 経由で履歴サムネ表示 フェーズ2
週次 / 月次グラフ カロリー・PFC・運動量の推移をビジュアライズ フェーズ2

3. 技術スタック

フロントエンド
Next.js 15 (App Router)

TypeScript / Tailwind CSS / Turbopack。Server Component で集計、Client Component で入力。

バックエンド / DB
Supabase

Postgres + Auth + Storage + RLS。サーバー/ブラウザ両クライアントを @supabase/ssr で実装。

AI 推論
Google Gemini

@google/genai SDK。responseSchema で構造化 JSON を強制。

デプロイ想定
Vercel

環境変数で API キー / モデル名を切り替え。プレビュー環境は別 Supabase プロジェクトを推奨。

⚠️ Gemini モデル名の注意: ユーザー指定の gemini-3.5-flash は現時点で公開モデル名として確認できません。 動作しない場合は GEMINI_MODEL=gemini-2.5-flash へ切り替えてください （コードは環境変数で完全に外出ししており再ビルド不要）。

7. 着手前にユーザーがやること

  • .env.local に GEMINI_API_KEY と Supabase 接続情報を記入
  • Supabase の SQL Editor で supabase/schema.sql を実行
  • npm run dev でローカル起動 → 写真1枚で解析動作を確認
  • モデル名が通らない場合は GEMINI_MODEL=gemini-2.5-flash へ変更

8. 次フェーズの候補 (優先度順)

  1. Supabase Auth 導入 — 1人用から複数ユーザー対応へ。RLS は既に有効化済。
  2. トレーニング記録 — workouts / sets テーブル追加 + 入力画面。
  3. 消費カロリーの推定 — トレーニング内容 + 体重から MET 法で算出、ダッシュボードに「収支」表示。
  4. 写真の Storage 保存 — 履歴にサムネ表示、振り返り体験を強化。
  5. グラフ化 — Recharts などで週次/月次の推移を可視化。
